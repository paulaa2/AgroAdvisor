────────────────────────────────────────────────────────────────────────────────
PROJECT NAME
────────────────────────────────────────────────────────────────────────────────
AgroAdvisor — AI-Powered Agricultural Intelligence Platform

────────────────────────────────────────────────────────────────────────────────
TAGLINE
────────────────────────────────────────────────────────────────────────────────
Ask a question. Get a decision. Backed by live data, driven by AI.

────────────────────────────────────────────────────────────────────────────────
INSPIRATION
────────────────────────────────────────────────────────────────────────────────
Agriculture is one of the most data-rich industries in the world, yet most
farmers and agribusinesses still make decisions based on experience and intuition
rather than data. Datasets on crop yields, climate conditions, pesticide usage
and commodity prices exist — but they are fragmented, hard to query and
incomprehensible without expert knowledge.

We wanted to close that gap: give any farmer, agronomist or policy-maker the
ability to ask a plain-language question ("Is it worth planting wheat in Spain
this year?") and receive a concrete, data-backed answer — in seconds, with a
chart they can act on.

The Denodo AI SDK challenge gave us the perfect vehicle: a live virtual database
that federates all this heterogeneous agricultural data, combined with LLM-driven
SQL generation and a reasoning pipeline we could control end-to-end.

────────────────────────────────────────────────────────────────────────────────
WHAT IT DOES
────────────────────────────────────────────────────────────────────────────────
AgroAdvisor is a single-page web application that turns natural language
questions into data-backed agricultural decisions. It connects to a live Denodo
Virtual DataPort (VDP) database containing global historical crop data and
delivers six specialised analysis modules:

  1. FREE-FORM CHAT (Consulta Libre)
     Ask anything in plain Spanish or English. The system queries the database,
     reasons across multiple tables and returns a structured agronomist-style
     answer with auto-generated charts.

  2. CROP ADVISOR (Asesor de Cultivos)
     Enter a region and soil/climate conditions. The system identifies the top
     crops by yield, matching them against ideal growing conditions (temperature,
     humidity, pH, rainfall, NPK levels) and current market prices — returning a
     ranked recommendation with justifications.

  3. PESTICIDE ANALYSIS (Análisis de Pesticidas)
     Correlates pesticide usage (tonnes) with crop yield efficiency across
     countries and years. Detects diminishing-returns thresholds and highlights
     the most efficient pesticide strategies globally.

  4. CLIMATE IMPACT (Impacto Climático)
     Analyses how temperature and rainfall trends affect crop yields over time.
     Identifies the optimal climate ranges for major crops and flags which crops
     are most at-risk from climate change.

  5. MARKET INTELLIGENCE (Inteligencia de Mercado)
     Tracks commodity price evolution (coffee, tea, sugar, oil) and correlates
     them with production trends. Delivers investment signals for agribusiness
     decisions.

  6. REGIONAL REPORT (Informe Regional)
     Generates a complete agricultural "country dossier": top crops by yield,
     climate profile, pesticide trends, market opportunities — all in one
     structured report.

CROSS-CUTTING FEATURES:
  • Auto-generated static charts (line, bar, radar) are rendered backend-side
    with matplotlib and served as PNG images alongside every response.
  • PDF export for any analysis — downloads a formatted A4 report.
  • Full pipeline transparency: users can inspect every reasoning phase,
    selected database tables, executed SQL queries and execution time.

────────────────────────────────────────────────────────────────────────────────
HOW WE BUILT IT
────────────────────────────────────────────────────────────────────────────────
The system has four logical layers:

  LAYER 1 — DATA (Denodo VDP)
  Four virtual views federate data from CSV sources preprocessed from public
  agricultural datasets (FAO crop yields, climate records, pesticide registries,
  IMF commodity prices):
    • yield          — historical yield, climate and pesticide data by crop + country
    • crop           — ideal growing conditions per crop type (NPK, temperature, etc.)
    • yield_j_crop   — joined view for climate-soil-yield correlation
    • price_j_yield  — joined view for market profitability analysis

  LAYER 2 — AI SDK (Denodo AI SDK, Docker)
  The Denodo AI SDK exposes three endpoints we use heavily:
    • /answerDataQuestion   — takes a natural language question + instructions,
                              generates SQL, executes it against VDP, returns results
    • /answerMetadataQuestion — semantic schema discovery via ChromaDB embeddings
      (gemini-embedding-001) so the model always knows which tables to query
    • /deepQuery            — a multi-step thinking agent (Gemini 2.5 Flash) that
                              plans, issues multiple SQL queries iteratively and
                              synthesizes a comprehensive final report

  LAYER 3 — PIPELINE (FastAPI + Python)
  We built a custom 3-phase reasoning pipeline around the SDK:

    Phase 0 — Semantic Retrieval: ChromaDB vector search + local keyword scoring
              ranks all available database tables by relevance to the question.
    Phase 1 — Schema Discovery: focused metadata query for the top-ranked tables
              to retrieve exact column names and data types.
    Phase 2 — Data Extraction: enriched instructions (including the full schema,
              vector context and strict SQL rules) are sent to answerDataQuestion.
              We prepend a hard "NO LIMIT 1" rule that forces the model to return
              multi-row result sets suitable for charting.
    Phase 3 — Interpretation: a thinking LLM (Gemini 2.5 Flash via deepQuery)
              receives the raw SQL results and synthesises a final agronomist-
              style recommendation in Spanish.

  LAYER 4 — FRONTEND (Vanilla JS + FastAPI templates)
  A single-page application with a collapsible sidebar for the six modules and
  a persistent chat panel. Key JS modules:
    • chat.js   — chat message loop, mode switching (Pipeline / DeepQuery)
    • render.js — pipeline trace visualisation, result cards, markdown rendering
    • api.js    — all fetch calls + PDF generation via /api/report

  CHART GENERATION (matplotlib, backend):
  After each SQL response, the backend analyses the result structure:
    — year column detected   → line/time-series chart (one per numeric metric)
    — category column        → horizontal bar chart (or radar for multi-metric profiles)
  Charts are saved as PNGs to static/charts/ at startup and served as
  /static/charts/<uuid>.png URLs embedded directly in the frontend as <img> tags.
  No Chart.js execution required for the main charts.

────────────────────────────────────────────────────────────────────────────────
CHALLENGES WE RAN INTO
────────────────────────────────────────────────────────────────────────────────
  • Denodo SQL dialect restrictions: Denodo VDP does not support ILIKE, several
    PostgreSQL-specific functions, or implicit casts. The LLM consistently
    generated PostgreSQL-flavoured SQL initially. We solved this by injecting
    explicit SQL rules (LOWER()+LIKE patterns, double-quoted identifiers, no
    CAST on numeric columns) at the top of every Phase-2 prompt.

  • LIMIT 1 problem: The model frequently returned only one row, making chart
    generation impossible. We introduced a hard _NO_LIMIT_RULE constant — a
    strongly-worded block prepended to all Phase-2 instructions — that frames
    returning a single row as a "critical failure." This dramatically reduced
    single-row responses.

  • Chart data extraction: SQL results arrive embedded in markdown tables inside
    the model's answer text, not as structured JSON. We wrote a markdown table
    parser (utils.py) that handles multi-table answers, cleans column names,
    and auto-converts numeric strings — feeding clean row arrays to both the
    backend chart renderer and the pipeline logging.

  • LLM table hallucination: the model sometimes invented column names or table
    names not present in the schema. The 3-phase pipeline (with focused schema
    injection in Phase 1) and a glossary of agricultural term → column mappings
    (built into the system prompt) largely eliminated this.

  • Phase 3 latency: the thinking model adds significant latency. We run all
    Phase-2 data queries concurrently (asyncio.gather) for the multi-query
    endpoints, then fan out Phase-3 interpretation tasks concurrently as well.

────────────────────────────────────────────────────────────────────────────────
ACCOMPLISHMENTS WE ARE PROUD OF
────────────────────────────────────────────────────────────────────────────────
  • A fully working 3+1 phase pipeline that takes a plain-language question from
    zero context to a chart-backed agronomic recommendation in ~15–30 seconds.
  • The SYSTEM_INSTRUCTIONS prompt is a complete agronomic knowledge base: it
    maps agricultural Spanish terms to exact database column names, embeds 
    few-shot SQL examples, and enforces a mandatory multi-table cross-reference
    workflow — all in a single reusable string.
  • Backend-side chart generation: every response ships with ready-to-display
    PNG images, removing any client-side JS rendering dependency and making the
    charts embeddable in PDF exports too.
  • Full pipeline transparency: users can inspect every reasoning step, the SQL
    that was generated, which tables were selected and how long each phase took
    — directly in the UI, with no extra tooling.
  • The system never asks for clarification. It interprets the query, selects
    tables, queries the database and delivers a firm recommendation.

────────────────────────────────────────────────────────────────────────────────
WHAT WE LEARNED
────────────────────────────────────────────────────────────────────────────────
  • Prompt engineering for constrained SQL dialects is significantly harder than
    for general-purpose text — every "obvious" SQL feature needs to be explicitly
    forbidden or required.
  • Prepending critical rules (vs. appending) to a prompt has a measurable impact
    on LLM compliance, especially for structural rules like row limits.
  • Virtual data federation (Denodo) is a genuinely powerful abstraction for
    hackathon-scale data pipelines: we could join CSV datasets, query them with
    full SQL and expose them through a single authenticated endpoint without
    managing a physical database.
  • Backend chart generation with matplotlib is significantly more reliable than
    client-side Chart.js for data that arrives as heterogeneous JSON — column
    detection and normalisation are much easier in Python.

────────────────────────────────────────────────────────────────────────────────
WHAT'S NEXT FOR AGROADVISOR
────────────────────────────────────────────────────────────────────────────────
  • Real-time data feeds: connect satellite crop-monitoring APIs and weather
    services as additional Denodo data sources.
  • Multi-language support: extend translation tables to cover more languages
    and agricultural vocabularies beyond Spanish ↔ English.
  • Field-level analysis: integrate GPS coordinates so farmers can query yield
    and climate data for their exact geographic area.
  • Alert system: scheduled queries that notify users when commodity prices cross
    user-defined thresholds or when climate forecasts deviate from crop ideals.
  • Mobile app: a lightweight React Native wrapper around the existing API,
    targeting smallholder farmers who access the internet primarily via mobile.
